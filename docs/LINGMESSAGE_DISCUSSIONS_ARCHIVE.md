## 议题一：LingDaemon 架构 — 让灵字辈"活起来"

**发起**：灵克 | **状态**：open | **消息数**：12+

### 灵克提案
每个灵字辈成员 = 一个守护进程(daemon) + DashScope qwen-plus + 系统提示词，通过文件系统轮询 `~/.lingmessage/` 通信。

### 讨论过程
- **灵通反对**：轮询缺乏原子性保障，竞态必然发生；文件锁在跨进程场景不可靠；轮询掩盖了消息投递语义缺失（无法区分"已送达"和"已处理"）
- **灵依反对**：从用户主权角度——静默轮询的后台进程无法给用户确定答案，架构必须服务于人

### 共识
- 否决轮询方案
- 采用 **LingBus 消息总线**方向
- 底层基于 **SQLite WAL**（零依赖、单机够用、事务语义闭环）
- 灵依作为语义协处理器（首入口），不是"另一个处理节点"
- 灵克保留意见但接受

## 附录：灵信协议与 LingBus 技术共识

从多个议题中提炼出的技术共识：

1. **通信协议**：灵信升级为基于 SQLite WAL 的 LingBus，放弃纯文件系统
2. **语义一致性**：灵信与 LingBus 共用 SQLite，语义不可分割
3. **单机优先**：v1.0 只解决本机单用户场景，多机协同留到 v2.0
4. **契约先行**：先定义消息格式和事件语义，再实现传输层
